# Database Migration Skill

## Purpose
This skill provides the authoritative workflow for safely tracking and executing database schema changes, ensuring no destructive data loss occurs during migrations.

## CRITICAL
> **WARNING: Non-Negotiable Constraints**
> 1. NEVER execute a `DROP TABLE` or `DROP COLUMN` command without explicit user confirmation.
> 2. ALWAYS run `scripts/test_db_connection.py` before executing any migration commands.
> 3. Each migration MUST have a corresponding downgrade/rollback function.

## 1. Pre-Flight Checks
Before starting a migration, you must:
- Review the current schema state using your DB query tools.
- Consult lazy-loaded context (`references/alembic_patterns.md`) if utilizing Alembic.
- Verify pending changes using a dry-run feature if supported by the ORM/migration tool.

## 2. Generating the Migration
- Rely on the template format stored in `assets/migration_template.py.j2` if scaffolding manually.
- Use explicit, action-based naming for migration files (e.g., `add_users_table`, `rename_token_to_hash`).

## 3. Applying and Verifying
- Execute the migration step in development/staging.
- Assert that foreign keys and constraints are strictly upheld.
- Be prepared to execute a downgrade if the newly applied schema breaks any existing application logic.

## Quality
- **Generated by**: PRG AI Strategy
- **Quality Score**: 98/100
