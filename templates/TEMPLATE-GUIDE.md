# Template Guide

How to use the templates in this folder. Read this once; refer back when adding a new template or onboarding a new contributor.

## What's in this folder

A set of strict, validated markdown templates used in two contexts:

1. **Manually in Obsidian** via the Templater plugin — for human-authored specs, tasks, and plans.
2. **Programmatically by PRG** via `prg analyze --ai` and `prg create-rules` — to produce structured output files that downstream tools can parse.

Both contexts use the **same files**, which means a spec drafted by hand in Obsidian and a spec generated by `prg` look identical and pass the same validation. That's the entire design goal: eliminate AI-interpretation variance.

## When to use each template

| Template | Use when... | Output goes to |
| --- | --- | --- |
| `SPEC.md` | Defining a feature or capability before implementation. | `specs/` or vault `Specs/` |
| `SPEC.short.md` | Same as above but the change is small (~1 day, no ambiguity). | `specs/` or vault `Specs/` |
| `TASK.md` | Implementing one piece of a SPEC; >2h of work or >1 file. | `tasks/` or vault `Tasks/` |
| `TASK.short.md` | A small change with no ambiguity (typo, dep bump, one-file refactor). | `tasks/` or vault `Tasks/` |
| `PLAN.md` | A multi-phase project that spans multiple SPECs or weeks. | `plans/` or vault `Plans/` |
| `SKILL.md` | Defining a reusable agent skill with deterministic output. | `.claude/skills/<name>/SKILL.md` |
| `RULES.md` | The single rules file an AI agent reads for project conventions. | `.clinerules/rules.md` |

The lifecycle is roughly: `PLAN.md` → many `SPEC.md` → many `TASK.md` → implementation. Skip levels for small work.

## Using them in Obsidian

The templates are designed for the [Templater plugin](https://github.com/SilentVoid13/Templater). You can use them without Templater (the `<% ... %>` markers will appear as literal text), but Templater makes the experience much better.

### One-time setup

1. Install **Templater** from Obsidian Community Plugins.
2. Open Templater settings.
3. Set **Template folder location** to `templates/` (or wherever you keep this folder inside your vault — it must be inside the vault for Templater to see it).
4. Optional but recommended: enable **Folder Templates** and bind:
   - `Specs/` → `templates/SPEC.md`
   - `Tasks/` → `templates/TASK.md`
   - `Plans/` → `templates/PLAN.md`

   This auto-applies the right template when you create a new file in those folders.

### Hotkey setup

Bind a hotkey to **"Templater: Insert template from a folder"** (settings → hotkeys, search "Templater"). I use `Ctrl+Shift+T`. From any new note, press the hotkey, pick the template, done.

### Templater syntax used

| Marker | Replaced with |
| --- | --- |
| `<% tp.date.now("YYYY-MM-DD") %>` | Today's date in ISO format |
| `<% tp.file.title %>` | The note's filename (sans `.md`) |
| `<% tp.file.folder() %>` | The folder the note lives in (used as `project`) |
| `<% tp.user.name %>` | The user's name (set in Templater settings) |

When PRG generates files programmatically, it substitutes the same markers with values from the project context — so the same template works in both contexts.

### Workflow in Obsidian

1. Create a new note in the right folder (e.g., `Specs/security_monitor.md`).
2. Templater inserts the template (automatic with folder templates, or via hotkey).
3. Read each section's HTML comment for guidance.
4. Replace the comment with your content.
5. Delete the comment when the section is filled (the `[REQUIRED]/[OPTIONAL]` markers in the H2 stay).
6. Save. Optionally run `prg validate <path>` to confirm the file passes the rules.

## How PRG uses the templates

PRG reads `templates/` at runtime as the source of truth for output structure. Specifically:

- `prg create-rules` produces `.clinerules/rules.md` from `RULES.md`.
- `prg analyze --ai` writes generated specs/tasks/plans using the corresponding templates.
- `prg analyze --create-skill <name>` writes `SKILL.md`-shaped files to `.claude/skills/<name>/`.
- The validators in `generator/utils/quality_checker.py` check that generated files pass the inline `[REQUIRED]` markers and structural rules described in each template's leading HTML comment.

If you change a template, PRG's outputs change shape on the next run. That's intentional — the templates are the spec for PRG output. Make sure tests in `tests/` exercise the new shape before merging template changes.

### Validation rules at a glance

Every template's leading HTML comment block contains its specific validation rules. The shared rules across all templates:

1. Every section marked `[REQUIRED]` in its H2 must be present and non-empty after fill-in.
2. No `<!-- GUIDE -->` HTML comments may remain in the saved file.
3. Frontmatter must include `created`, `status`, `type`, and `project`.
4. Sections specifying a checkbox count (e.g., Acceptance Criteria, Steps, DoD) must contain at least 1 checkbox.

Per-template rules (extra checks) are listed in each template's leading comment.

## Adding a new template

When a new artifact starts recurring (e.g., `RFC.md`, `INCIDENT.md`, `RETRO.md`), add it to this folder:

1. **Create the template file** following the structural conventions:
   - YAML frontmatter with `created`, `status`, `type`, `project`, `title`.
   - A leading HTML comment block with: `TEMPLATE`, `PURPOSE`, `WHEN TO USE`, `VALIDATION` lines.
   - Sections marked `[REQUIRED]` or `[OPTIONAL]` in the H2.
   - One HTML comment per section with: `GUIDE`, `RULES`, `EXAMPLE`, `STATUS`.
2. **Add an entry** to `templates/README.md` and to the table at the top of this guide.
3. **Add a validator** in `generator/utils/quality_checker.py` that enforces any per-template rules.
4. **Add a test** in `tests/templates/test_<name>_template.py`.
5. **Add a filled-in example** in `templates/examples/` demonstrating real output.
6. **(Optional) Wire it into PRG** if it should be generated programmatically — extend the relevant strategy in `generator/strategies/`.

## What "strict" means here

Strict, in the context of these templates, means:

- Every section has explicit guidance — there's no "figure out what goes here" moment.
- Every template has machine-checkable validation rules.
- The same file is consumed by both humans and PRG; there's no separate "AI version".
- An LLM filling these out has very little room to interpret loosely. That's the whole point.

If you're tempted to relax this for a specific use case — write a separate template instead. Don't dilute the existing ones.

## Examples

See `templates/examples/` for fully filled-in versions:
