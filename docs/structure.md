# Output Structure

Project Rules Generator consolidates all generated files into a single `.clinerules/` directory inside your project. This keeps your root directory clean and makes it easy to manage rules.

```
.clinerules/
├── rules.md              # Main coding rules (DO/DON'T, testing, etc.)
├── constitution.md       # High-level principles (optional, enabled by --constitution)
├── clinerules.yaml       # Lightweight YAML skill references for Cline
├── skills/
│   ├── builtin/          # Copied builtin workflow skills
│   │   ├── test-driven-development.md
│   │   ├── systematic-debugging.md
│   │   └── ...
│   ├── learned/          # Project-specific patterns (AI-generated)
│   │   └── api-optimization.md
│   └── index.md          # Skills catalog/registry
└── .prg-cache.json       # Cache for incremental mode (git-ignored)
```

## File Descriptions

-   **`rules.md`**: The core rules file. Contains critical "DOs" and "DON'Ts", testing instructions, dependency information, and architectural overview. This is always generated.
-   **`constitution.md`**: A higher-level document focusing on coding philosophy and long-term principles. Generated only when `--constitution` flag is used.
-   **`clinerules.yaml`**: A configuration file used by the generator and potentially other tools to understand active skills and project settings.
-   **`skills/`**: A directory containing all active skills for the project.
    -   **`builtin/`**: Skills that come with Project Rules Generator (e.g., TDD, debugging).
    -   **`learned/`**: Custom skills generated by the AI specifically for your project (e.g., specific API usage patterns).
    -   **`index.md`**: A master list of all available skills.
-   **`.prg-cache.json`**: An internal cache file used to speed up incremental updates. It should be added to `.gitignore`.
