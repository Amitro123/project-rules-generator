# PRG Skills vs Anthropic Skill Spec — Full Comparison & Improvement Guide

After reading the official Anthropic Skills Guide and cross-referencing it with your entire skills mechanism, here are every gap, every alignment, and every concrete improvement you can make.

## ✅ Where PRG Already Aligns

Before diving into gaps, PRG does several things correctly:

- **`SKILL.md` as the canonical file** — matches Anthropic's required file name (case-sensitive)
- **Directory-based skill structure (`skill-name/SKILL.md`)** — matches Anthropic's folder-per-skill pattern
- **3-layer resolution (project > learned > builtin)** — mirrors Anthropic's scoping concept
- **Trigger extraction via `SkillParser`** — concept is correct, implementation needs work (see below)
- **Quality gating via `validate_quality()` + `is_stub()`** — PRG has this, Anthropic doesn't (PRG is actually ahead here)
- **Strategy fallback chain (AI → README → Cowork → Stub)** — a PRG original, not in the spec but a solid design

## ✅ GAP 1 — YAML Frontmatter Schema Mismatch (DONE)

**Anthropic spec** — every `SKILL.md` must open with YAML frontmatter:

```yaml
---
name: fastapi-development        # kebab-case, matches folder name
description: |
  Generates FastAPI routes, Pydantic schemas, and async handlers.
  Use when user asks to "add endpoint", "create API route",
  or mentions FastAPI, Pydantic, or async API patterns.
license: MIT
allowed-tools: "Bash(python:*) Bash(uvicorn:*)"
metadata:
  author: PRG
  version: 1.0.0
  category: backend
  tags: [fastapi, python, async, api]
---
```

**PRG today** — `SKILL.md` files are generated as pure Markdown. Triggers are stored in a separate `auto-triggers.json` file, which Claude does not natively read. The YAML frontmatter is absent from generated skills.

**Impact:** Skills generated by PRG are not natively loadable in Claude.ai or Claude Code as first-class skills. They work only when the agent reads the whole `.clinerules/` directory.

**Fix:**

```python
# In CoworkSkillCreator.create_skill() / _render_skill_md()
def _render_frontmatter(self, metadata: SkillMetadata) -> str:
    trigger_phrases = ", ".join(f'"{t}"' for t in metadata.auto_triggers[:5])
    desc = (
        f"{metadata.description}. "
        f"Use when user mentions {trigger_phrases}."
    )
    return (
        f"---\n"
        f"name: {metadata.name.replace('_', '-')}\n"
        f"description: {desc[:1024]}\n"
        f"allowed-tools: \"{' '.join(metadata.tools)}\"\n"
        f"metadata:\n"
        f"  author: PRG\n"
        f"  version: 1.0.0\n"
        f"  category: {metadata.category}\n"
        f"  tags: [{', '.join(metadata.category.split())}]\n"
        f"---\n\n"
    )
```

## ✅ GAP 2 — Naming Convention Violation (DONE — was already kebab-case)

**Anthropic spec:** folder and skill names must be strictly kebab-case. No underscores, no capitals, no spaces. `fastapi-api-development`, not `fastapi_api_development`.

**PRG today** — `TECH_SKILL_NAMES` dictionary:

```python
TECH_SKILL_NAMES = {
    "fastapi": "fastapi_api_development",   # ❌ underscores
    "react":   "react_component_development", # ❌ underscores
    "pytest":  "pytest_testing",              # ❌ underscores
    ...
}
```

**Impact:** Skill folders created with underscores won't be resolved by Claude.ai's native skill loader.

**Fix:** A single utility + bulk rename:

```python
# generator/utils/naming.py
def to_kebab(name: str) -> str:
    return name.lower().replace("_", "-").replace(" ", "-")

# Apply to TECH_SKILL_NAMES:
TECH_SKILL_NAMES = {
    "fastapi": "fastapi-api-development",
    "react":   "react-component-development",
    "pytest":  "pytest-testing",
    ...
}
```

## ✅ GAP 3 — No Progressive Disclosure (3-Level Structure) (DONE)

**Anthropic spec** defines three levels:

| Level | What | When loaded |
|-------|------|-------------|
| 1 | YAML frontmatter | Always (system prompt) |
| 2 | `SKILL.md` body | When skill is deemed relevant |
| 3 | `scripts/`, `references/`, `assets/` | On-demand as agent navigates |

**PRG today:** Every skill is a single flat `SKILL.md` — all content is at level 2. There is no per-skill `scripts/` for runnable code, no `references/` for on-demand docs, no `assets/` for templates.

**Impact:** PRG skills are documentation-only. They can tell Claude what to do but can't include executable validation scripts or reference templates that get loaded progressively.

**Fix** — add scaffold generation to `TaskCreator.create_skill()`:

```text
.clinerules/skills/learned/fastapi-development/
  SKILL.md              ← Level 1 (frontmatter) + Level 2 (instructions)
  scripts/
    validate.py         ← Level 3: run to verify the skill works
    generate_route.py   ← Level 3: executable helper
  references/
    pydantic-patterns.md   ← Level 3: loaded on demand
  assets/
    route_template.py.j2   ← Level 3: Jinja2 template
```

## ✅ GAP 4 — Trigger Description Format Doesn't Follow Spec (DONE)

**Anthropic spec** — the description field must follow this exact template:

`[What it does] + [When to use it — explicit trigger phrases]`

*Good example:*
"Analyzes Figma design files and generates handoff docs. Use when user uploads .fig files, asks for 'design specs', 'component documentation', or 'design-to-code handoff'."

*Bad examples (Anthropic explicitly flags these):*
"Helps with projects." ← too vague
"Implements the FastAPI endpoint model." ← technical/internal language, no user triggers

**PRG today:** `_derive_project_skills()` and `CoworkSkillCreator._build_metadata()` build descriptions from tech context extraction — they produce technical descriptions, not user-facing trigger phrases.

**Fix** in `CoworkSkillCreator._build_metadata()`:

```python
def _format_description(self, meta: SkillMetadata) -> str:
    trigger_str = ", ".join(f'"{t}"' for t in meta.auto_triggers[:4])
    return (
        f"{meta.description} "
        f"Use when user mentions {trigger_str}, "
        f"or asks about {meta.name.replace('-', ' ')} patterns."
    )[:1024]
```

## ✅ GAP 5 — No Negative Triggers (DONE)

**Anthropic spec:** recommends adding negative triggers to the description to prevent over-triggering. Example: "Do NOT activate for general Python questions unrelated to FastAPI endpoints."

**PRG today:** `auto-triggers.json` only stores positive trigger phrases. There's no mechanism to prevent a skill from loading on loosely related queries.

**Fix** — add `negative_triggers` to `SkillMetadata`:

```python
@dataclass
class SkillMetadata:
    ...
    auto_triggers: List[str] = field(default_factory=list)
    negative_triggers: List[str] = field(default_factory=list)  # ← new

# In description builder:
neg_str = ", ".join(f'"{t}"' for t in meta.negative_triggers[:3])
if neg_str:
    desc += f" Do NOT activate for {neg_str}."
```

## 🟡 GAP 6 — Quality Scoring is Static, Not Behavioral (Medium)

**Anthropic spec** quality target:
- ≥ 90% trigger rate on relevant queries
- 0 failed API calls per workflow
- Users complete tasks without follow-up prompts

**PRG today** — `validate_quality()` scores by:
- ✅ Required sections present
- ✅ Trigger count ≥ N
- ✅ Tool names in content
- ✅ Minimum length
- ✅ Numbered steps
- ❌ No trigger precision test (does the skill actually fire on the right queries?)
- ❌ No A/B comparison (with vs without skill)

**Fix** — add a trigger-precision evaluator:

```python
# generator/utils/trigger_evaluator.py
class TriggerEvaluator:
    """Test trigger precision: should-fire vs should-not-fire queries."""
    
    def evaluate(self, skill_md: str, test_cases: List[TriggerTestCase]) -> TriggerReport:
        """
        Returns precision score: how many 'should_fire' queries
        contain at least one trigger phrase from the skill.
        Target: >= 0.9
        """
        triggers = self._extract_triggers(skill_md)
        hits = sum(
            1 for tc in test_cases
            if tc.should_fire == self._matches_any(tc.query, triggers)
        )
        return TriggerReport(precision=hits / len(test_cases), total=len(test_cases))
```

## 🟡 GAP 7 — Resolution is Exclusive, Not Composable (Medium)

**Anthropic spec:** Claude can load multiple skills simultaneously from the system prompt. Skills should be designed to coexist and not assume they're the only one active.

**PRG today:** `SkillDiscovery.list_skills()` returns one skill per name (project > learned > builtin resolution). `resolve_skill()` returns a single path. Skills are not designed to be composed.

**Fix** — add a `resolve_active_skills()` method:

```python
def resolve_active_skills(self, query: str) -> List[Path]:
    """Return ALL skills whose triggers match the query (composable)."""
    triggers_map = self.skills_manager.extract_all_triggers()
    active = []
    query_lower = query.lower()
    for skill_name, triggers in triggers_map.items():
        if any(t.lower() in query_lower for t in triggers):
            path = self.resolve_skill(skill_name)
            if path:
                active.append(path)
    return active
```

### Quality Checker Fixes (bonus)
Two bugs fixed in `quality_checker.py` during this work:
- Section parser used `.split("##")` which also split on `###` sub-headers → fixed to `re.split(r"\n## ", ...)`
- Step count regex `r"^\s*\d+\."` didn't match Jinja2 template's `### N.` format → fixed to `r"(?:^\s*\d+\.|^#{2,3}\s+\d+\.)"`

### TriggerEvaluator Design Note
`auto_test_cases()` skips negative trigger test cases that contain a positive trigger phrase, because simple substring matching would always fire on them (e.g. `"general fastapi questions"` contains `"fastapi"`). Negative triggers are semantic hints for the LLM and require embedding-based evaluation for accurate precision testing.
